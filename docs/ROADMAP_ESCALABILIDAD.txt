PRINCIPIOS CLAVE:
════════════════════════════════════════════════════════════════════════════

1. START SIMPLE, SCALE WHEN PAIN POINTS APPEAR
   No optimices para problemas que no tienes todavía

2. MEASURE BEFORE OPTIMIZING
   Logging de fallos → datos reales → mejoras dirigidas

3. PROGRESSIVE ENHANCEMENT
   Keywords → Embeddings básicos → FAISS → Hybrid
   Cada paso con trigger claro

4. COST/BENEFIT ANALYSIS
   2 horas keywords (+5%) >> 1 día embeddings (+8%) para 2K normas
   Pero a 10K normas: embeddings >> keywords

5. AVOID OVER-ENGINEERING
   ❌ NO necesitas Pinecone con 2K normas
   ❌ NO necesitas GPU con <20K normas
   ❌ NO necesitas microservicios con <10 usuarios


RIESGOS DE SOBRE-INGENIERÍA (NO HACER AHORA):
════════════════════════════════════════════════════════════════════════════

❌ Vector databases (Pinecone, Weaviate, Qdrant)
   Razón: Overkill para 2K vectores, numpy es más rápido

❌ Fine-tuning custom de modelos
   Razón: Necesitas >50K normas para justificar

❌ Arquitectura microservicios
   Razón: Complejidad innecesaria para <10 usuarios

❌ GPU dedicada
   Razón: CPU suficiente para 2-20K normas

❌ Multiple modelos ensemble
   Razón: Mantenimiento complejo, ganancia marginal <3%

❌ Real-time updates
   Razón: Batch processing mensual es suficiente
